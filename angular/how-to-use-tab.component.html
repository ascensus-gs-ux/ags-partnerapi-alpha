SSO & authentication

Partner SSO uses the OAuth 2.1 confidential-client model with the private_key_jwt authentication method (RFC 7523 JWT bearer profile). No shared secrets are exchanged.

Authentication model

  • Confidential client over TLS — your application authenticates with a signed JWT, not a client secret.
  • Both assertions are signed with the same X.509 key pair registered for your application.
  • Assertions are short-lived, single-use, and scoped to one environment — generate a new jti (UUID) per request.

SSO flow

  • Browser-based session model — SSO is carried by the user's authenticated browser session.
  • No re-authentication between the partner and an embedded MFE within the same session.
  • Sessions are browser-bound and end when the browser session ends.

Partner setup

  • Requires a registered application and certificate with Ascensus.
  • Authentication uses an X.509 key pair with the private_key_jwt method.
  • Keys are managed through the Admin Portal.

Token requirements

  • Each request must present a unique JWT — generate a new jti (UUID) every time.
  • Tokens are short-lived; iat and exp are enforced.
  • Reuse of a JWT fails authentication.
Error & failure conditions
  • Expired or reused JWT → authentication fails.
  • Incorrect environment → token rejected.
  • Missing proxy routing → session fails.
  • Freeze code or invalid account status → request blocked.

Restrictions & requirements

Account-level conditions that determine whether a request can proceed.

Important rules
  • Only status 91 (active) accounts can be modified.
  • Freeze codes block transactions until the restriction is removed.
  • Web-disabled accounts prevent all API activity.
Account Status ⌄

Only accounts with status 91 can be modified via PUT.

Web Access Disabled ⌄

When enabled, all activity is blocked. Confirm webDisabled is false before proceeding.

Freeze Codes ⌄
